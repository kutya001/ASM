# Архитектура и Стек (Четкие правила разработки)

Этот документ описывает технологический стек, файловую структуру и **жесткие правила**, которые необходимо соблюдать. Следуя им, можно создать аналогичную CRM под любую другую бизнес-логику (например, для салонов красоты или клиник).

## 🛠 Технологический Стек

- **Фреймворк**: Vue.js 3 (Composition & Options API)
- **Сборщик**: Vite
- **Стейт-менеджер**: Pinia
- **Стилизация**: Tailwind CSS (PostCSS)
- **Язык**: JavaScript (с элементами JSDoc/поддержкой типов)
- **Иконки**: Bootstrap Icons

## 📂 Структура проекта

```text
src/
├── assets/          # Статические стили и изображения
├── components/
│   ├── layout/      # Обертки (Header, Sidebar, MobileNav)
│   ├── modals/      # Модальные окна (ProfileModal, RecordModal, RefModal) 
│   ├── common/      # Переиспользуемые мелкие UI элементы
│   └── ui/          # Базовые UI-компоненты (опционально)
├── games/           # Игротека (HillClimb, FlappyBird и др.)
├── store/           # Pinia хранилище (index.js, queue.js)
├── utils/           # Утилиты (audioHelper.js, formatters.js)
├── views/           # Экраны/Вкладки (RecordsTab, DashboardTab, UsersTab)
├── App.vue          # Корневой компонент
└── main.js          # Точка входа, подключение плагинов
```

## 🏗 Архитектурные паттерны и правила (Универсальные)

Чтобы проект легко развивался, каждый разработчик обязан следовать правилам ниже:

### 1. Архитектура хранилища и синхронизации
**Правило:** Запрещено вызывать HTTP-запросы (fetch/axios) напрямую из компонентов (Vue) для мутации данных!
- **Чтение:** Все данные должны грузиться при старте в глобальное состояние (Pinia). Компоненты читают данные из стора.
- **Мутация & Оптимистичный UI:** Если пользователь добавляет запись, компонент вызывает действие стора `store.dispatchSync('addRecord', data)`. 
- **Очередь (SyncQueue):** Хранилище немедленно применяет изменение локально для отрисовки, а сетевой запрос складывается в очередь. Таким образом, бизнес-логика (например, CRM поликлиники) потребует только описать новые типы в сторе и эндпоинты в `dispatchSync`.

### 2. Модальные окна
**Правило:** Модальные окна не монтируются внутри списков через `v-for`. 
- Модалки подключаются **один раз** в корне (App.vue) или в крупной обертке. Различные кнопки вызывают метод `.open(data)` через `$refs.recordModal.open(...)`, передавая туда объект на редактирование. 
- Это предотвращает создание сотен скрытых DOM-элементов при длинных списках.

### 3. Структура Компонента
- Внутри компонента запрещена сложная фильтрация, занимающая много строк. Все переиспользуемые/сложные выборки либо выносятся в `<script>` секцию в `computed: {}`, либо в `Pinia getters`.
- Компоненты-вкладки (Views: RecordsTab, RefsTab) не хранят состояние сессии. Они лишь отрисовывают то, что передано в пропсах от корневого `App.vue` или стейта напрямую. 

### 4. Дизайн-система (Tailwind CSS)
**Правило:** Использование кастомных классов в `<style>` секциях запрещено или резко ограничено.
- Цвета, скругления, тени должны соответствовать системным утилитам Tailwind (например, `bg-slate-50`, `rounded-xl`, `border-slate-200`).
- Избегайте использования сторонних UI библиотек (Vuetify, ElementUI), чтобы не раздувать бандл и не ломать единый дизайн-код Tailwind. Вся верстка реализована самостоятельно.

### 5. Обработка ошибок (Toast-сообщения)
**Правило:** Локальный `alert()` запрещен.
Все ошибки (валидация полей, сетевые отклонения) должны выводиться через глобальный массив тостов. Для этого используется экшен хранилища `showToast('Текст', 'error')`.

### 6. Мобильная адаптивность (Mobile - First)
Интерфейсы должны изначально выглядеть работоспособно при ширине 320px-480px.
- Использовать `md:flex-row`, `md:hidden` для изменения компоновки на десктопах.
- Мобильная навигация (`MobileNav.vue`) используется как нижнее меню (Bottom Bar), которое скрывается на `md`.

### Как запустить новый проект с этой же архитектурой?
1. Взять структуру `App.vue`, модуль `store` и компоненты `Layout`, `Modals`.
2. Заменить схему данных (например, `Records` на `Appointments`, `Brands` на `Doctors`).
3. Заменить поля в форме сохранения.
4. Вся подкапотная работа по очередям, стейту, фильтрам и аутентификации будет уже работать.
