# Бизнес-логика приложения

В данном документе описаны процессы, статусы и регламенты, которые формируют ядро AutoCRM. Система была специально спроектирована под нужды автомастерских.

## 1. Ролевая модель
В системе предусмотрено разделение доступа по ролям (Role):
- **Superadmin (Владелец / Администратор):** 
  - Видит все заявки.
  - Управляет пользователями системы.
  - Может удалять данные, проводить "массовые обновления".
  - Управляет справочниками (отрасль авто, услуги, марки и модели).
- **Master (Мастер / Исполнитель):**
  - Видит записи, закрепленные только за ним (опредляется по `MasterID`).
  - Может изменять статусы своих записей.
  - Не может редактировать справочники.
  - Имеет ограниченный интерфейс приложения.

## 2. Сущности системы

- **Record (Запись):** Центральная сущность. Каждая запись содержит:
  - Имя клиента, номер телефона, гос. номер автомобиля (`CarNumber`).
  - Привязку к модели и марке из справочников (Связь через `BrandID`, `ModelID`).
  - Список оказанных услуг (сериализованный JSON `ServicesJSON`, например `["Замена масла", "Диагностика"]`).
  - Статус (Открыт, Выполнен, Отменён).
  - Привязанного Мастера (`MasterID`).
  - Финансовый статус (Оплачено — `IsPaid`).

- **Services (Услуги):** Справочник с наименованиями работ, ценами, нормо-часами. Записи ссылаются на услуги по названиям/ID.
- **Brands / Models (Марки и Модели):** Двухуровневый справочник. Без выбора марки нельзя отфильтровать ее модели. 
- **Application (Новая Заявка):** Отдельная очередь "карантина". Заявка приходит сверху, администратор ее обрабатывает и либо переводит в `Record` (назначает мастера и открывает наряд), либо отменяет.

## 3. Жизненный цикл заявки (Записи)
Стандартный бизнес-процесс для Записи описывается конечным автоматом статусов:

1. **Создание:** Регистрация новой заявки. Ей присваивается начальный статус: **"Открыт"**. Запись закрепляется за конкретным мастером (либо остается без мастера, если только планируется).
2. **В работе:** Автомобиль прибыл, мастер получает уведомление. 
3. **Завершение:** Мастер переводит статус в **"Выполнен"**. При этом система *автоматически* проставляет время завершения (`EndTime = new Date().toISOString()`), фиксируя время работы. Мастер не может изменять оплаченные/закрытые записи.
4. **Отбой:** Если клиент не приехал, статус меняется на **"Отменён"**.

## 4. Оплата (Менеджмент финансов)
Запись независимо от ее статуса выполнения обладает флагом `IsPaid` (Оплачено).
- **Superadmin** может снимать и устанавливать галочку.
- **Master** может отметить запись как "Оплаченную" (передал деньги в кассу или лично получил на карту), но *не может* изменять финансовый статус у закрытых (Выполненных) заявок. Это защита против мошенничества.

## 5. Система геймификации
Любой труд утомляет, поэтому добавлена система мини-игр. 
Внутриигровая валюта (Coins / Gems) может быть синхронизирована с продуктивностью мастера. Например: 1 выполненная заявка = 100 Ruby (рубинов). Рубины могут быть потрачены на разблокировку новых скинов для байка в игре `Rider` или `HillClimb`. 
(На данный момент логика вознаграждения реализована частично локально `localStorage`, готовится интеграция с бэкендом).
