# Спецификация для реализации аналогичного проекта (CRM-система)

Данная спецификация описывает архитектурные, интерфейсные и технические требования для создания аналогичного приложения (CRM-системы) с использованием **идентичного стека и подходов**, но адаптированного под **иную бизнес-логику** (например, управление салоном красоты, клиникой, ремонтной мастерской электроники и т.д.). В данной спецификации исключены модули геймификации (Игротека).

## 1. Технологический стек

В основе проекта лежит современный и производительный стек технологий, обеспечивающий быструю разработку, удобную масштабируемость и высокую скорость работы приложения в браузере.

- **Фреймворк:** Vue.js 3 (Composition API / Options API).
  - *Vue 3* выбран за его эффективную систему реактивности (на базе Proxy) и высокую производительность виртуального DOM.
  - Использование *Composition API* позволяет гибко переиспользовать сложную бизнес-логику (композаблы управления очередями, формами, сетью) вне компонентов, делая архитектуру более модульной.
  - Допускается использование *Options API* для простых презентационных компонентов, чтобы не переусложнять код.
- **Сборщик:** Vite.
  - Сборщик нового поколения, который обеспечивает мгновенный запуск dev-сервера и сверхбыструю "горячую" замену модулей (HMR), что в разы ускоряет процесс UI-разработки.
  - Для продакшен-сборки Vite использует оптимизированный под капотом Rollup, генерируя легковесные и оптимально разделенные (code-splitting) бандлы.
- **Управление состоянием (State Management):** Pinia.
  - Официальный, легковесный и модульный стейт-менеджер для Vue (заменяет устаревший Vuex). 
  - Обеспечивает интуитивно понятное управление состоянием без лишней "лапши" из мутаций. Каждая доменная область (Users, Records, Auth) имеет свой независимый стор.
  - Является "Единым источником правды" (Single Source of Truth) и играет якорную роль в реализации механизма "Оптимистичного UI" и очереди фоновой синхронизации.
- **Стилизация:** Tailwind CSS (PostCSS).
  - *Утилитарный подход (Utility-first)*: позволяет верстать компоненты прямо в HTML-шаблонах без создания десятков дополнительных CSS файлов (используя классы вроде `bg-white`, `px-4`, `rounded-xl`).
  - Гарантирует строгое соблюдение дизайн-системы. Палитра цветов, отступы и размеры шрифтов жестко заданы в конфигурации, что предотвращает визуальный хаос на проекте.
  - Работает поверх *PostCSS*, автоматически удаляя неиспользуемые стили (Purge/Tree-shaking) при финальной сборке, что сводит размер стилей для загрузки к минимуму.
- **Иконки:** Bootstrap Icons.
  - Универсальная, строгая и легкая библиотека векторных иконок, идеально подходящая под деловой стиль CRM-интерфейсов. Легко масштабируется и стилизуется с помощью утилит цвета Tailwind.

## 2. Пользовательский интерфейс (UI/UX) и Layout

Проект должен сохранять "чистый" интерфейс, ориентированный на мобильные устройства (Mobile-First) и быструю работу "в поле". 

### 2.1. Глобальная структура (Layout)

Конструкция интерфейса строится на базе двухмерной сетки с фиксированными слоями навигации, адаптирующейся под размеры экрана устройства. Ключевая концепция — разделение рабочей области на независимые панели с прокруткой для сохранения фокуса и предотвращения прокрутки всей страницы (`overflow-hidden` на уровне корневого элемента `body`).

#### 2.1.1. Общая сетка и Органы управления
- **Корневой контейнер (`App.vue` или главный Layout):**
  - Должен иметь фиксированную высоту `h-screen` (или `h-[100dvh]` для мобильных браузеров, исключая полосы прокрутки встроенного интерфейса Safari/Chrome).
  - Сетка десктопа: двухколоночная разметка на базе Flexbox (`flex flex-row overflow-hidden w-full h-full`).
  - Фоновый слой всего приложения декорируется мягким цветом `bg-slate-50` для повышения контрастности белых карточек контента.

#### 2.1.2. Sidebar (Левая навигационная панель для десктопа)
Появляется только при ширине экрана от `md` (`hidden md:flex flex-col`).
- **Размеры и позиционирование:**
  - Фиксированная ширина `w-64` или `w-72` для предотвращения горизонтального сжатия контента.
  - Высота `h-screen`, жесткие рамки за счет правой границы `border-r border-slate-200 bg-white`.
- **Внутренняя структура холста:**
  - **Верхний слот (Header/Brand Area):** Высота ориентировочно `64px`, содержит логотип компании, название системы и версию.
  - **Центральный слот (Navigation Menu):** `flex-1 py-4 px-3 space-y-1 overflow-y-auto`. Ссылки навигации должны подстраиваться под роли пользователя (напр., прятать вкладки работы со справочниками и пользователями для роли `Master`).
    - *Состояния ссылок:* Элементы меню имеют скругления `rounded-lg` или `rounded-xl`. Активное состояние подсвечивается (`text-indigo-600 bg-indigo-50/70`), неактивное имеет цвет `text-slate-600 hover:bg-slate-50 hover:text-slate-900 transition-all duration-200`.
  - **Нижний слот (Profile & Actions Slot):** Закреплен в самом низу боковой панели. Содержит карточку авторизованного пользователя (Имя, скрытая роль через tooltip) и кнопку безопасного выхода из сессии (`Logout`).

#### 2.1.3. Header (Верхняя панель управления контекстом)
- **Позиционирование:**
  - Размещается в верхней части рабочей области основного экрана (`flex flex-col flex-1 h-full overflow-hidden`).
  - По высоте фиксирован в пределах `h-16` (64px) с нижней обводкой `border-b border-slate-200 bg-white/80 backdrop-blur-md` (эффект матового стекла).
- **Элементы управления:**
  - Левая часть: кнопка переключения видимости меню для промежуточных планшетных разрешений или отображение текущей даты и приветственного статуса.
  - Прямо по центру или в правой части: модуль мгновенного поиска (поиск заявок/мастеров/клиентов) и блок точечных уведомлений (колокольчик активности).
  - Крайняя правая точка: мини-аватар или селектор быстрой смены активного филиала (при масштабировании).

#### 2.1.4. Основная рабочая область (Main Viewport)
- **Контейнер для вкладок:**
  - Занимает всю оставшуюся область под Header: `flex-1 overflow-y-auto bg-slate-50`.
  - В фоновом режиме контейнер оборачивается во вспомогательный ограничитель ширины (`w-full max-w-7xl mx-auto px-4 sm:px-6 lg:px-8 py-6`), предотвращая визуальное растягивание на сверхшироких мониторах (UHD/4K).
- **Компоновка контента:**
  - Для каждой страницы (Дашборд, Список записей) макет выстраивается по правилу единого ритма полей. Наружные поля и карточки разделяются равными промежутками `space-y-6`.
  - Внутри вкладок рекомендуется использовать структуру "белых карточек" (`bg-white rounded-2xl border border-slate-200/80 shadow-sm`).

#### 2.1.5. MobileNav (Нижняя навигационная панель для мобильных устройств)
Инициализируется и рендерится при ширине экрана строго менее `md` (`md:hidden flex`).
- **Спецификации размещения:**
  - `position: fixed; bottom: 0; left: 0; right: 0; z-index: 40; height: 64px;`
  - Фон: полностью непрозрачный белый `bg-white` или с размытием заднего плана `bg-white/95 backdrop-blur-lg` с верхней направляющей границей `border-t border-slate-150` и эффектом тени `shadow-[0_-4px_12px_rgba(0,0,0,0.03)]`.
  - Обязательная поддержка отступа под системный индикатор современных безрамочных систем (iPhone/Android Notch) — использование паддинга `pb-[env(safe-area-inset-bottom,0)]`.
- **Сетка и Лимиты кликов:**
  - Равномерное табличное или флекс-распределение кнопок по всей ширине (`grid grid-cols-4` или `flex justify-around items-center h-full`).
  - Минимальная зона клика для каждой кнопки: `44px` в высоту и ширину (согласно стандартам Apple Human Interface Guidelines).
  - **Метки и иконки:** По центру каждой ячейки располагается векторная иконка (`text-lg`), а прямо под ней — лаконичный текстовый ярлык (`text-[10px] font-medium leading-tight`), подсвечивающийся активным цветом (`text-indigo-600`) при совпадении текущего роута/вкладки. В неактивном состоянии цвет иконки сглажен до нейтрального серого (`text-slate-400`).

### 2.2. Модальные окна

Управление диалоговыми окнами и формами ввода является критической частью CRM, так как в них происходит 90% манипуляций с данными. Для обеспечения безупречной производительности и отсутствия задержек на мобильных устройствах, проектирование модальных окон должно строго следовать трем архитектурным принципам.

#### 2.2.1. Паттерн Единого Экземпляра (Singleton Modal UI Pattern)
- **Суть паттерна:** Каждое уникальное интерактивное окно (например, `RecordModal` — форма записи, `RefModal` — редактирование справочника) должно существовать в дереве DOM **строго в единственном экземпляре**.
- **Локация монтирования:** Все модальные компоненты импортируются и рендерятся один раз в шаблоне корневого корневого контейнера (`App.vue` или `Layout.vue`) вне зависимости от структуры текущих открытых вкладок.
- **Программный вызов:** 
  - Взаимодействие и передача данных в модальное окно осуществляются через механизм ссылок Vue (`ref`). К примеру, при клике на строку таблицы записи, кнопка вызывает метод корневого уровня:
    `this.$refs.recordModal.open(selectedRecordData)`.
  - Метод `open(data)` внутри самого компонента модального окна инициализирует внутреннее реактивное состояние формы на основе переданного объекта (или очищает его, если создается новая запись), переводит внутренний флаг видимости в `true` и блокирует прокрутку заднего плана страницы (`document.body.classList.add('overflow-hidden')`).
- **Категорический запрет на v-for:** Ни в каком виде недопускается монтирование модальных окон внутри итерируемых циклов списка (`v-for`). Создание сотен скрытых инстансов тяжелых форм ввода критически нагружает оперативную память мобильного браузера, увеличивает размер виртуального DOM-дерева и замедляет отрисовку интерфейса.

#### 2.2.2. Логика Списков и Безопасный Скролл (Safe Touch Scroll Dropdowns)
- **Проблема свайпов на мобильных устройствах:** При просмотре длинных выпадающих списков выбора (таких как справочники брендов или услуг) внутри модального окна, пользователь совершает вертикальные свайпы по экрану для прокрутки. 
- **Критический баг с mousedown/touchstart:** Если на элементы списка повесить обработчики клика через события `@mousedown.prevent` или `@touchstart.prevent` (часто использующиеся разработчиками для исключения потери фокуса с инпутов), выбор элемента сработает мгновенно в момент первого соприкосновения пальца с экраном. Таким образом, любая попытка скролла списка вниз приведет к мгновенному случайному выбору и закрытию выпадающего меню.
- **Архитектурное решение:** 
  - Замена всех тяжелых событий тача и мыши на стандартные клик-обработчики: `@click.stop="selectItem(item)"`.
  - Мобильные операционные системы и веб-киты нативно умеют гасить и блокировать события `click`, если они распознают жест прокрутки (pan/scroll). Использование `@click` с модификатором `.stop` гарантирует, что прокрутка списка пальцем будет работать гладко, а выбор сработает только тогда, когда пользователь совершил короткий прицельный тап без продольного смещения пальца.

#### 2.2.3. Защита от Утери Данных и Предотвращение Дублирования (UI Guards)
- **Интерактивные барьеры на закрытие:** При клике на фоновую подложку модального окна (backdrop), событие клика на корневом оверлее не должно автоматически закрывать форму, если пользователь уже начал заполнять поля ввода. 
  - *Реализация:* При триггере клика по бэкдропу компонент сверяет текущее состояние полей с исходным (флаг `isDirty`). Если изменения найдены, закрытие блокируется и выводится предупреждающее диалоговое окно, либо фоновое закрытие полностью отключается, заставляя пользователя явно нажать на крестик `"Закрыть"`.
- **Защита от двойной отправки (Double-Submit Prevention):** Кнопка сохранения (`Сохранить` / `Подтвердить`) должна сопровождаться индикатором загрузки. 
  - В момент отправки формы в стор, кнопке присваивается свойство `disabled`, а ее текстовое содержимое заменяется на анимированный спиннер. Это предотвращает повторные клики (спам-клик) по кнопке во время выполнения фоновых микрозадач или медленной сетевой транзакции, исключая дублирование транзакций в базе данных.

### 2.3. Визуальный стиль
- Цветовая палитра выстраивается на основе стандартов Tailwind (например, базовые фоновые цвета — `slate-50`, `white`, акценты — на основе `indigo` и т.д.).
- Отказ от использования сторонних тяжеловесных UI-библиотек (Vuetify, Element UI). Верстка компонентов (кнопки, инпуты, селекты, таблицы) выполняется вручную с использованием утилитарных классов Tailwind CSS.
- Глобальная и централизованная система уведомлений (Toasts) для отображения ошибок, а не нативные функции `alert()`.

## 3. Архитектура и Логика синхронизации (Оптимистичный UI)

Главная особенность проекта — способность работать быстро, создавая ощущение нулевой задержки (Zero Latency) и возможность функционировать в условиях нестабильного интернета за счет фоновой синхронизации.

### 3.1. Хранилище (Принцип Single Source of Truth)

Централизованное управление состоянием на базе Pinia выступает в роли единственного достоверного источника правды (Single Source of Truth) для всего приложения. Компоненты UI полностью избавлены от задач прямого выполнения запросов, кеширования и сложной фильтрации данных.

#### 3.1.1. Разделение ответственности и запрет на локальный стейт данных
- **Запрет на локальное дублирование:** Компоненты Vue не должны хранить локальные копии массивов данных, полученных с сервера (например, копировать `this.records = store.records` в хуке `mounted`). Это приводит к рассинхронизации интерфейса и потере реактивности при фоновых обновлениях. 
- **Потребление данных через геттеры:** Любые выборки, фильтрации и группировки (например, фильтрация записей по активному мастеру или дате) должны выполняться через вычисляемые свойства компонентов (`computed`), которые напрямую ссылаются на `getters` в Pinia. Это гарантирует мгновенную перерисовку UI при любых реактивных изменениях в сторе.
- **Однонаправленный поток данных (Unidirectional Data Flow):** Данные спускаются от Стора к компонентам по реактивным связям. Любые изменения (создание, удаление, редактирование, перевод статуса оплаты) должны инициироваться исключительно путем вызова действий (actions) в соответствующем сторе.

#### 3.1.2. Стандартизованная структура стейта (Store State Schema)
Каждый доменный стор (например, `records`, `users`, `services`) должен иметь строго определенную и предсказуемую структуру состояния:
- `items`: Основной массив сущностей в оперативной памяти.
- `isLoading`: Булевый флаг общего процесса загрузки данных (используется для отображения глобальных скелетонов/спиннеров загрузки и блокировки интерфейса).
- `isInitialized`: Флаг, указывающий на то, что первичное наполнение стора данными (из кэша локального хранилища или сервера) завершено. Предотвращает дублирующие запросы при переходах между вкладками.
- `lastSyncTime`: Временная метка последней успешной синхронизации с бэкендом (используется для предотвращения повторных тяженых запросов в короткий промежуток времени).

#### 3.1.3. Шаблон взаимодействия "Компонент -> Стор"
Любая мутация данных в интерфейсе компонента должна быть строго декларативной. Компонент сигнализирует о намерении совершить изменение, передавая полезную нагрузку (payload) в Стор.

*Пример реализации на стороне компонента:*
```javascript
// Внутри Vue-компонента (например, кнопка перевода статуса записи)
methods: {
  async completeRecord(recordId) {
    this.isActionPending = true;
    try {
      // 1. Компонент вызывает экшен стора, передавая только ID и целевые параметры
      await this.recordsStore.updateRecordStatus({
        id: recordId,
        status: 'Completed'
      });
      this.showToast('Запись успешно завершена!', 'success');
    } catch (error) {
      this.showToast('Не удалось завершить запись', 'error');
    } finally {
      this.isActionPending = false;
    }
  }
}
```

#### 3.1.4. Логика мутаций внутри Стора
Внутри экшена стора выполняется разделение на локальное (оптимистичное) обновление состояния и регистрацию задачи синхронизации:
1. **Валидация:** Первичная проверка структуры и корректности отправляемых данных.
2. **Локальный коммит (Commit):** Прямая мутация локального массива `items` в оперативной памяти для моментального переключения флагов в UI.
3. **Логирование в SyncQueue:** Передача задачи в менеджер очередей для её отложенной трансляции на сервер.

### 3.2. Очередь Синхронизации (SyncQueue)
В основе изменения данных лежит механизм очереди:
1. **Действие:** Пользователь нажимает "Сохранить" в форме.
2. **Оптимистичное обновление (Optimistic Update):** Хранилище *немедленно* обновляет локальные данные (добавляет или редактирует запись). Пользовательский интерфейс перерисовывается моментально, без ожидания ответа сервера.
3. **Регистрация задачи в Очереди:** Хранилище формирует объект задачи (например, `{ action: 'CREATE_RECORD', payload: { ... } }`) и помещает его в глобальную очередь (`SyncQueue`).
4. **Фоновая сеть:** Фоновый процесс очереди прослушивает наличие задач и выполняет реальные HTTP-запросы (fetch / axios / WebSocket) на сервер последовательно.
5. **Обработка результатов:**
   - *Успех:* Задача удаляется из очереди.
   - *Ошибка доступа к сети:* Задача остается в очереди со статусом "Ожидание". Очередь будет пытаться повторить отправку позже при восстановлении связи.
   - *Критическая ошибка (например, валидация на бэкенде или превышение лимитов):* Локальный стейт откатывается (Rollback), пользователю выводится Toast-сообщение о неудаче.

### 3.3. Локальное хранение (Offline Support)
- Данные справочников и локальная очередь должны кэшироваться в браузере (`LocalStorage` или `IndexedDB`).
- При загрузке приложения первоначально восстанавливается стейт из локального хранилища, затем осуществляется фоновая сверка актуальных данных с сервером.

## 4. Адаптация под другую Бизнес-Логику

Чтобы развернуть проект под **иную доменную область**, необходимо переопределить следующие модули, сохранив само ядро нетронутым (Синхронизацию, методы API в Store, компоненты Layout):

### 4.1. Изменение Сущностей (Entities)
Задаются новые базовые сущности и их поля.
*Пример: Стоматологическая Клиника*
- **Records (Приемы/Записи):** Дата, Пациент (Имя, Телефон), Исполнитель (Врач, MasterID), Услуги (Лечение кариеса, Консультация), Диагноз/Жалобы, Статус (Запланировано, В процессе, Завершено, Отменено).
- **Refs (Справочники):**
  - Вместо Марки/Модели —> Категории услуг, Классификаторы.
  - Прайс-лист услуг клиники.
- **Users (Работники):** Врачи (вместо автомехаников), Администраторы ресепшена.

### 4.2. Ролевая модель
Механизм доступа остается неизменным, меняются только названия и границы контекста:
- **Admin / Manager:** Доступ ко всем данным, отчетность, финансы, управление персоналом и настройка справочников.
- **Staff / Executor (Врач, Парикмахер и т.д.):** Ограниченный доступ исключительно к своим назначенным "Записям" (на сегодня/неделю). Механизмы самостоятельного изменения статусов (Ожидает -> В кресле -> Завершено).

### 4.3. Views (Экраны)
- **Дашборд:** Графики адаптируются под актуальные метрики бизнеса (Количество клиентов, Выручка за день, Популярные услуги).
- **Таблица Записей:** Колонки таблицы изменяются (например, "Пациент" вместо "Модель Авто", "Жалобы" вместо "Пробег").
- **Справочники:** Экраны CRUD (создание, редактирование таблиц) переименовываются под нужные объекты бизнеса.

## 5. Исключения оригинального проекта (Геймификация)
Текущая версия спецификации подразумевает классический Enterprise/B2B подход. 
Проект **не должен** содержать раздел `Игротека` (`games/` или любую другую развлекательную механику). 
Компоненты canvas-игр, логика накопления локальной или виртуальной валюты (монеты/кристаллы), рейтинговые таблицы игроков лобового стекла и турниры — **полностью исключаются** из архитектуры и кода, с целью сохранения строгого фокуса на выполнении CRM и ERP задач организации.
